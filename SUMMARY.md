## Root cause

Two `stablehlo.reduce_window` variants are generated by the model that are not handled
by the existing `StableHLOToTTIRReduceWindowOpConversionPattern` in
`tt-mlir/lib/Conversion/StableHLOToTTIR/StableHLOToTTIRPatterns.cpp`.

**Primary failure (`reduce-window.112`)**: `torch.cummax` at
`modeling_mm_grounding_dino.py:1767` (`prev_special = torch.cummax(prev_special, dim=1)`)
is lowered by XLA to a 2-output `stablehlo.reduce_window` with:
- Input: `tensor<1x9xi64>` (values), `tensor<1x9xi32>` (indices)
- Window: `[1, 9]` with left padding `[[0, 0], [8, 0]]`
- Body: `compare(GE) + select` for max value; `equal + select` for tie-breaking index
- Output: `(tensor<1x9xi64>, tensor<1x9xi32>)` — two outputs

TTIR has `ttir.argmax` (full-reduction argmax, not windowed) and
`ttir.max_pool2d_with_indices` (2D float, not 1D integer). Neither covers a 1D integer
windowed argmax. No `ttir.cummax` op exists.

**Secondary failures (`reduce-window.730`, `reduce-window.762`, etc.)**:
`pixel_mask.cumsum(1, ...)` and `pixel_mask.cumsum(2, ...)` at lines 1714-1715 are on a
rank-3 `pixel_mask` tensor of shape `[batch, height, width]`. These produce rank-3
add-body reduce_windows; the existing pattern handles rank-1 cumsum only.

The loader also had four independent bfloat16 dtype mismatch bugs (all fixed):
1. `spacy` namespace collision causing `datasets` import crash — replaced with `PIL.Image.new`
2. `get_text_position_embeddings` returning float32 pos embedding that promotion-casts bfloat16 queries
3. `MultiScaleDeformableAttention.forward` — `grid_sample(bf16_value, f32_grid)` dtype mismatch
4. Cascading float32 sources (`get_sine_pos_embed`, `generate_encoder_output_proposals`,
   `valid_ratios = valid_ratios.float()`) feeding bfloat16 decoder paths

All four were patched in the loader. The compiler fails only after the loader is clean.

## Fix

**Loader fixes applied** (tt_forge_models,
branch `remediation/mm_grounding_dino-pytorch-Tiny_O365V1_GoldG_V3Det-single_device-inference`):
- `mm_grounding_dino/pytorch/loader.py`: 4 commits adding `_patch_mm_grounding_dino_dtype()`
  and `_register_valid_ratios_hooks()` plus switching `load_dataset` to `PIL.Image.new`

**Compiler fix required** (not implemented): In
`tt-mlir/lib/Conversion/StableHLOToTTIR/StableHLOToTTIRPatterns.cpp`, the
`StableHLOToTTIRReduceWindowOpConversionPattern` must be extended or new patterns added:
1. Recognize a 2-output reduce_window with `compare(GE)+select` / `equal+select` body as
   `cummax`, and lower it to a TTIR `cummax` op (which does not yet exist) or decompose it.
2. Extend the existing 1D cumsum reduce_window pattern to rank-3 tensors.

Adding `ttir.cummax` requires changes in at minimum:
- `include/ttmlir/Dialect/TTIR/IR/TTIROps.td` — new op definition
- `lib/Conversion/StableHLOToTTIR/StableHLOToTTIRPatterns.cpp` — recognition + lowering
- `lib/Conversion/TTIRToTTNN/` — TTNN lowering
- Backend kernel support in `tt-metal`

This crosses more than 3 files and spans multiple repos.

## Tier B justification
**Indicator: new-infrastructure**. TTIR has no `cummax` (windowed argmax) operation for
1D integer inputs. Adding one requires new op definition (`TTIROps.td`), StableHLO→TTIR
lowering, TTIR→TTNN lowering, and backend kernel support across at least 4 files in 2 repos.
The multi-output reduce_window body (2 outputs, integer comparison) also has no analogue in
the existing max_pool / avg_pool / cumsum patterns.
